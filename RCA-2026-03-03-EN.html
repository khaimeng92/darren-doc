📋 Executive Summary

The MongoDB → Pulsar → ClickHouse sync pipeline in PMS2.0_PH suffered a cascading failure that generated 270 million duplicate consumer records and 185 uncontrolled process crashes over ~24 hours on 2026-03-03. The resume token (checkpoint) was never saved — it grew from 13.2 hours stale to 35.6 hours stale by end of day, causing each restart to replay progressively more historical data.

The root cause is a one-line bug in generateUniqueTimestamp() in syncProposals.js: the documentId parameter is accepted but never used in the hash. When a batch contains multiple documents sharing the same proposalId and transactionType, no iteration can ever resolve the collision — the fixDuplicateTimestamps while-loop spins forever. Log evidence confirms it reached 6,044,274 iterations before the first crash. This problem existed as early as 2026-03-02.

🔍 Root Cause Analysis

Root Cause #1 (Primary — First Domino): generateUniqueTimestamp ignores documentId

In syncProposals.js, the function signature accepts documentId but the hash computation excludes it entirely:

// syncProposals.js — generateUniqueTimestamp
export const generateUniqueTimestamp = (completeTime, proposalId, transactionType, documentId, iteration = 0) => {

    const uniqueStr = `${proposalId}_${transactionType}_${iteration}`;
    // ↑ documentId is received but NEVER USED in the hash — the bug

    const hashCode = uniqueStr.split('').reduce((a, b) => {
        a = ((a << 5) - a) + b.charCodeAt(0);
        return a & a;
    }, 0);
    const ms = Math.abs(hashCode % 1000);
    return `${baseTime}.${ms.toString().padStart(3, '0')}`;
};

Consequence: Two documents with the same proposalId + transactionType (different _id) produce identical timestamps at every iteration. The fixDuplicateTimestamps while-loop can never terminate for such pairs:

// syncClickhousePluginConsumer.js
let iteration = 0;
while (fixDuplicateTimestamps(mappedDocs, populatedDocs, processId, ++iteration)) {
    // docs with same proposalId+transactionType → same hash at EVERY iteration
    // → always returns true → loop never exits
    // → observed in production: 6,044,274 iterations before OOM crash
}

Root Cause #2: Three-Layer Token Persistence Failure

Even if the process restarts gracefully, the resume token is never reliably saved due to three compounding defects:

#DefectEffect
AshutdownChangeStreams is not exported from syncClickhousePlugin.js — consumer's shutdown() destructs an undefined functionToken save silently skipped on every graceful shutdown
Bprocess.on('exit', asyncFn) cannot await async operationsRedis write starts but process exits before completion

Log confirmed: "Saved resume tokens" = 0 occurrences across all 185 restarts. The Mar 2 12:26 token was loaded every single time.

Root Cause #3: Silent Error Swallowing in Producer .catch

.catch((error) => {
    console.error('changeStream | Producer for clickhouse sync failed...', error);
    // ❌ _markDocumentAsSent() never called → doc eligible for re-send on next restart
})

Root Cause #4: Consumer ACKs Before ClickHouse Write

All Pulsar messages are acknowledged immediately upon receipt, before processBatch() runs. Write failures result in permanent data loss from Pulsar's perspective.

🔧 Fix Plan

🚨 P0 — Fix generateUniqueTimestamp: include documentId in hash (1-line fix)

// syncProposals.js
const uniqueStr = `${proposalId}_${transactionType}_${iteration}`;
const uniqueStr = `${proposalId}_${transactionType}_${documentId}_${iteration}`;
// MongoDB _id is always unique → no two documents can produce the same hash

🚨 P0 — Add max iteration guard to while-loop

const MAX_DEDUP_ITERATIONS = 10;
let iteration = 0;
while (iteration < MAX_DEDUP_ITERATIONS && fixDuplicateTimestamps(mappedDocs, populatedDocs, processId, ++iteration)) {
    if (iteration >= MAX_DEDUP_ITERATIONS) {
        console.error(`[${processId}] fixDuplicateTimestamps hit max iterations — breaking to prevent infinite loop`);
        break;
    }
}

🚨 P0 — Periodic resume token save (every 30 seconds)

With this fix, a crash causes at most 30 seconds of replay instead of 35 hours.

// syncClickhousePlugin.js — inside _startCleanupInterval()
setInterval(async () => {
    await _saveAllResumeTokens();
}, 30 * 1000);

🚨 P0 — Export shutdownChangeStreams (fix graceful shutdown token save)

// syncClickhousePlugin.js — add export
export const shutdownChangeStreams = async () => {
    await _saveAllResumeTokens();
    for (const [, cs] of globalChangeStreams) await cs.close();
    globalChangeStreams.clear();
    _stopCleanupInterval();
};
// Also remove: process.on('exit', gracefulShutdown) — async on 'exit' cannot await

✅ Conclusion

This incident was caused by a single unused parameter in a hash function. generateUniqueTimestamp accepted documentId but excluded it from computation, making timestamp deduplication permanently impossible for documents sharing a proposalId and transactionType. The while-loop spun 6 million times, crashed the process, loaded a 13-hour-stale checkpoint, and replayed progressively more data on each of 185 subsequent crashes — growing the stale window to 35.6 hours and generating 270 million duplicate messages.

The fix is one line. The prevention — periodic token persistence — limits any future crash impact to 30 seconds of replay instead of hours.

Key Takeaway: A parameter accepted but not used is a silent contract violation. In uniqueness-guarantee functions, verify every input is actually consumed. Always cap retry loops with a hard maximum. Never rely solely on process-signal handlers for critical state persistence.
