Drop ClickHouse — Replace with Optimized MongoDB Analytics

Eliminate the entire ClickHouse sync pipeline by leveraging the existing MongoDB analytics replica with compound indexes and pre-aggregated summaries.

1. Executive Summary

The current system maintains a complex sync pipeline (MongoDB → Change Streams → Pulsar → Consumer + Populate → ClickHouse) to replicate 4 collections for reporting. This pipeline suffers from high MongoDB read load (double-reads via fullDocument: 'updateLookup' and heavy .populate() calls), persistent deduplication bugs, and operational complexity across 5+ scheduler jobs, a Pulsar consumer, and a dedicated PM2 worker.

The key insight is that the MongoDB-based reports already exist and work. The codebase has a separate analyticDB connection pointing to a dedicated MongoDB analytics replica, with *Analytic models for all 4 collections, and complete aggregation pipelines under server/src/components/report/. The ClickHouse report endpoints under reportClickhouse/ are a parallel copy of the same logic rewritten in SQL.

Proposal

Drop ClickHouse entirely. Optimize the existing MongoDB analytics replica with compound indexes and pre-aggregated summary collections. Reroute all /clickhouse-* endpoints to the existing MongoDB report handlers via a feature flag. Remove the entire sync pipeline.

2. Current Architecture & Problems

2.1 Current Data Flow (Dual Path)

CURRENT: Two parallel reporting paths

MongoDB Primary → Change Streams → Pulsar Queue → Consumer + Populate → ClickHouse → /clickhouse-report/*
MongoDB Primary → Schedulers (1–10 min) → ClickHouse
MongoDB Primary → Analytics Replica (built-in sync) → /report/*

2.2 Problems with the ClickHouse Path

4. Proposed Architecture

PROPOSED: Single path via MongoDB analytics replica

MongoDB Primary → Analytics Replica (built-in replication, ~seconds lag) → Unified Reports /report/* and /clickhouse-report/*
Nightly Pre-Aggregation Cron (02:00) → Summary Collections (payment/deposit/merchant _duration_daily)

Before (ClickHouse Path)

  • Change streams with fullDocument: 'updateLookup'
  • Pulsar queue + consumer
  • Heavy .populate() for denormalization
  • 5 scheduler jobs (1–10 min intervals)
  • ClickHouse ReplacingMergeTree with dedup bugs
  • Separate PM2 worker process
  • Minutes of sync lag + data loss risk

After (MongoDB Analytics)

  • Built-in replica sync (~seconds lag)
  • No Pulsar, no consumer, no change streams
  • No populate needed (queries join at read time via $lookup or pre-existing pipelines)
  • 1 nightly pre-aggregation cron
  • Zero dedup issues (replica is exact copy)
  • No extra PM2 workers
  • Compound indexes on analytics replica only

5. Optimization 1: Compound Indexes on Analytics Replica

Zero Impact on Writes

All indexes below are created only on the analytics replica (PMS_ANALYTIC_DB_URL). They do not affect write performance on the primary MongoDB instance.

The existing single-field indexes are insufficient for the report aggregation patterns. These compound indexes match the $match stages in the report pipelines:

6. Optimization 2: Pre-Aggregated Summary Collections

For the heaviest reports (duration bucket analysis across months), create nightly pre-aggregation jobs that write daily summaries to dedicated collections. This replaces ClickHouse materialized views.

6.1 Summary Collections

CollectionSourceGroup ByMetrics
payment_duration_daily proposalWithdrawals (analytic) date, minuteCategory, platform, type count, totalAmount, autoMatchCount
deposit_duration_daily proposalDeposits (analytic) date, minuteCategory, platform, paymentMethod, api count, totalAmount
merchant_duration_daily proposalMerchants (analytic) date, minuteCategory, platform, device count, totalAmount

6.2 Duration Buckets

Matching the existing ClickHouse multiIf / MongoDB $switch logic:

// Duration buckets in minutes
const DURATION_BUCKETS = [1, 2, 3, 5, 8, 10, 15, 20, 30, 60];
// Category 61 = over 60 minutes

// MongoDB $switch equivalent:
{ $switch: {
    branches: DURATION_BUCKETS.map(min => ({
        case: { $lte: ["$durationMs", min * 60000] },
        then: min
    })),
    default: 61
}}

6.3 Nightly Cron Schedule

A single scheduler job runs at 02:00 daily, aggregates the previous day's data from raw collections, and upserts into summary collections using bulkWrite with upsert: true.

// Scheduler config
const schedulerRule = { hour: 2, minute: 0 };
runScheduler(runPreAggregation, {
    schedulerRule,
    maxExecutionTimeInSec: 600,
    jobName: 'preAggregateDurationSummaries'
});

Reports check the summary collection first for historical data; fall back to raw aggregation for today's (not yet summarized) data.

8. Route Migration Strategy

8.1 Feature Flag

Add USE_MONGODB_REPORTS to server/src/config/env.js:

SKIP_CLICKHOUSE_SYNC: true,   // disable change streams + Pulsar
USE_MONGODB_REPORTS: true,    // route /clickhouse-* to MongoDB handlers

8.2 Router-Level Handler Selection

Modify each ClickHouse router to delegate to the MongoDB handler when the flag is enabled. This keeps the same URL paths so no client-side changes are needed:

// Example: reportClickhouse/paymentDuration/paymentDurationReportRouter.js
import ChHandler from "./paymentDurationReportHandler";
import MongoHandler from "../../report/paymentDuration/paymentDurationReportHandler";
import env from "../../../config/env";

const selectHandler = (chHandler, mongoHandler) => (req, res, next) => {
    const handler = env.config().USE_MONGODB_REPORTS ? mongoHandler : chHandler;
    return handler(req, res, next);
};

router.route('/clickhouse-report/payment-duration')
    .get(auth, selectHandler(ChHandler.getStatistic, MongoHandler.getStatistic));

11. Implementation Phases

Phase 1: Compound Indexes + Benchmark Low Risk

Duration: 1–2 days

  1. Create compound indexes on the analytics replica (idempotent, zero impact on primary)
  2. Run benchmark script comparing MongoDB report results vs ClickHouse report results for correctness and latency
  3. Validate that MongoDB reports return equivalent data

Files created:

  • server/src/scripts/createAnalyticIndexes.js — one-time migration script
  • server/src/scripts/benchmarkReports.js — comparison script

Phase 2: Pre-Aggregated Summaries Low Risk

Duration: 2–3 days

  1. Create nightly pre-aggregation cron job for the 3 duration summary collections
  2. Add to PM2 jobs config
  3. Backfill historical summaries for the past 6 months

Files created:

  • server/src/scheduler/reportPreAggregation/preAggregateDurationSummaries.js

Files modified:

  • server/pm2-jobs.config.js — add preAggregateDurationSummaries entry

Phase 3: Route Migration with Feature Flag Medium Risk

Duration: 2–3 days

  1. Add USE_MONGODB_REPORTS feature flag to env.js
  2. Modify 9 ClickHouse router files to use selectHandler pattern
  3. Test with flag false (no change) then flip to true
  4. Monitor report accuracy and latency in staging

Files modified:

  • server/src/config/env.js
  • 9 ClickHouse router files (see Section 8.3)

Phase 4: Disable ClickHouse Sync Medium Risk

Duration: 1 day

  1. Set SKIP_CLICKHOUSE_SYNC: true in env config
  2. Comment out syncClickhousePluginConsumer in pm2-jobs-2.config.js
  3. Comment out pulsarSyncOldProposalWorker in pm2-worker-syncClickhouse.config.js
  4. Remove .plugin(syncClickhousePlugin, ...) from 4 model schemas
  5. Remove direct Pulsar publish in transactionAction.js

Phase 5: Full Cleanup After Validation

Duration: 1 day (after 2+ weeks of stable operation)

  1. Remove all ClickHouse sync files (see Section 9)
  2. Remove @clickhouse/client from package.json
  3. Remove clickHouseClient.connect() from schedulerRunner.js
  4. Remove ClickHouse report endpoint files (or keep as dead code reference)
  5. Remove feature flag checks (hardcode MongoDB path)

12. Risk Mitigation

RiskMitigation
MongoDB reports are slower than ClickHouse for some queries Compound indexes + pre-aggregated summaries bring most queries under 10s. For any outlier, add a specific pre-aggregated collection rather than restoring the entire ClickHouse pipeline.
Analytics replica falls behind under heavy write load MongoDB replica lag is typically <1 second. Monitor with rs.printSlaveReplicationInfo(). The current ClickHouse pipeline has minutes of lag, so this is strictly better.
Feature flag rollback needed Set USE_MONGODB_REPORTS: false to instantly revert all endpoints to ClickHouse handlers. No deployment needed if using Apollo/Vault config.
Index creation impacts analytics replica reads Use background: true option. Run during low-traffic hours. Indexes are created once.
Pre-aggregation cron fails Reports fall back to raw aggregation (slower but correct). Cron failures are logged and can be re-run manually.

Sync lag Minutes (Pulsar + consumer batching) ~Seconds (built-in replica sync)
Deduplication issues Persistent (generateUniqueTimestamp defeats ReplacingMergeTree) None (replica is exact copy)
Infrastructure components MongoDB + Pulsar + ClickHouse + 5 schedulers + 2 PM2 workers MongoDB + 1 nightly cron

Bottom Line

Trade a small increase in report query latency (2–15s vs 1–5s) for: zero dedup bugs, zero sync lag, massively reduced MongoDB primary load, elimination of Pulsar/ClickHouse infrastructure, and a single nightly cron replacing 5 scheduler jobs + 1 Pulsar consumer + 2 PM2 workers.
