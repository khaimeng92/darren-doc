1. Background & Purpose

PMS uses MongoDB as the operational database for real-time transactions and proposal management. A copy of completed and in-progress proposals is synced to ClickHouse (columnar OLAP database) for reporting purposes, primarily querying monthly records for payment duration reports, deposit/withdrawal analytics, and merchant performance.

Data is pre-joined (denormalized) before insertion into ClickHouse. This means reference data such as platform names, bank names, merchant gateway codes, and user names are resolved at sync time via Mongoose .populate() and stored as flat columns. This is the correct approach—ClickHouse is optimized for scanning wide, flat tables and performs poorly on JOINs.

Problem
The current sync implementation is excessively expensive on MongoDB, causes duplicate records in ClickHouse, and has reliability issues with message acknowledgment. This spec proposes a refactoring to address all three.

2.2 Populate Chains (MongoDB Read Amplification)

Each .populate() call triggers additional MongoDB queries. Here is the cost per collection:

Collection Populate Depth Collections Touched Reads per Document
proposalWithdrawal 3 levels deep bankCard → merchantGroup → merchantGateway, bankType, cardSupplier, platform ×2, bankType, user ×2 ~8 reads
proposalMerchant 1 level merchantGateway, merchantGroup, platform ×2, playerRank ~5 reads
proposalDeposit 2 levels deep cardSupplier, bankCard → bankType, platform ×2, cardDeposit ~6 reads
transaction 1 level user ×2, platform, cardSupplier ~4 reads
Key Insight
For a batch of 1,000 withdrawal proposals, the consumer executes approximately 8,000 MongoDB reads (1 find query + 7 populate queries per document). The change stream already performed a read via fullDocument: 'updateLookup', making the total ~9,000 reads for 1,000 documents.

3. Identified Problems

3.1 Double-Read from MongoDB

syncClickhousePlugin.js (line 178: fullDocument: 'updateLookup') → syncClickhousePluginConsumer.js (line 190: .find().populate())

The change stream requests fullDocument: 'updateLookup', which forces MongoDB to perform an additional collection read on every update event to return the full document. However, only _id and syncTableName are sent to Pulsar—the full document is discarded. The consumer then re-reads the same document with a heavy .populate() chain.

Cost: Every document is read from MongoDB at least twice, with the second read amplified by 4-8x through populate.

3.2 Redundant Transaction Cron Job

syncTransactions.js (every 1 minute, up to 50,000 records)

The syncTransactions cron runs every minute and queries MongoDB by time range with .populate(). This is fully redundant with the change stream consumer, which already syncs completed transactions. The completed-status cron jobs for withdrawal, merchant, and deposit are already commented out—but the transaction one was not.

Cost: Up to 50,000 × 4 = 200,000 unnecessary MongoDB reads per minute.

3.3 Duplicate Records in ClickHouse

Duplicates originate from four sources:

SourceMechanismWhy Dedup Fails
Change stream + Cron overlap Both insert the same completed proposal No cross-system dedup; both insert independently
Change stream + Direct Pulsar publish transactionAction.js publishes to Pulsar on insert; change stream also fires on the same insert In-memory dedup Map only covers the change stream process
Multiple change stream events per document Status transitions (PENDING → PROCESSING → SUCCESS) each fire an event 5-min dedup window may expire between transitions; Map lost on PM2 restart
PM2 worker restart In-memory sentDocuments Map is cleared All previously tracked documents become "unsent"

3.4 ReplacingMergeTree Dedup Defeated by Timestamp Manipulation

syncProposals.js (line 304: generateUniqueTimestamp) → syncClickhousePluginConsumer.js (line 221: fixDuplicateTimestamps)

ClickHouse's ReplacingMergeTree deduplicates rows with the same ORDER BY key during background merges. If the ORDER BY key includes completeTime, and the code manipulates milliseconds to make timestamps unique, then:

  • The same document inserted by the consumer and the cron job will have different generated timestamps
  • ReplacingMergeTree sees them as different rows and keeps both
  • The fixDuplicateTimestamps function ensures uniqueness within a batch but creates divergence across batches
Critical
The timestamp manipulation actively prevents ClickHouse from deduplicating rows. This is the root cause of persistent duplicates in report queries.

3.5 Acknowledge-Before-Process Anti-Pattern

syncClickhousePluginConsumer.js (lines 91-96)

Messages are acknowledged immediately after receiving, before any processing occurs. If processing fails (ClickHouse down, mapping error, etc.), the message is permanently lost from Pulsar. The republishing mechanism that was meant to handle this is disabled (console.log('We removed the republishing')).

Risk: Data loss on any processing failure.

4. Proposed Architecture

4.1 Design Principles

  1. Read MongoDB once per document — eliminate the double-read
  2. Cache reference data — platform, bank, user names change rarely; don't query them per-document
  3. Use stable business keys for dedup — proposalId (or composite key for transactions), not timestamps
  4. Single source of truth per sync type — no overlapping paths for the same data
  5. Acknowledge after process — no data loss

4.2 New Data Flow

PATH 1: Change Stream → Pulsar → Consumer (Completed Proposals) ┌────────────────────────────────────────────────────────────────────────────────────┐ │ │ │ MongoDB Change Stream (NO updateLookup) │ │ │ │ │ │ filter by operationType + updateDescription.updatedFields │ │ ↓ │ │ Send { _id, syncTableName } to Pulsar (60s delay) │ │ │ │ │ ↓ │ │ Consumer (every 5s): │ │ 1. batchReceive() from Pulsar │ │ 2. Group by syncTableName, deduplicate by _id │ │ 3. model.find({ _id: $in }).lean() ←← SINGLE READ, no populate │ │ 4. Resolve references from in-memory cache │ │ 5. Map to ClickHouse format (no timestamp manipulation) │ │ 6. bulkInsert into ClickHouse │ │ 7. Acknowledge messages AFTER successful insert │ │ (negativeAcknowledge on failure → Pulsar auto-retries) │ │ │ └────────────────────────────────────────────────────────────────────────────────────┘ PATH 2: Cron Scheduler — Incomplete Proposals (every 10 minutes, KEEP) syncProposalWithdrawal.js → CANCEL, EXPIRED, PENDING, PROCESSING, etc. syncProposalMerchant.js → PENDING, EXPIRED syncProposalDeposit.js → PROCESSING, VERIFIED, EXPIRED, CANCEL, CLAIMED Modified: .find().lean() without populate, resolve from cache PATH 3: Daily Safety-Net Audit (NEW, replaces transaction cron) Runs once per day, compares ClickHouse counts vs MongoDB counts Re-syncs any gaps found in the last 48 hours REMOVED: Direct Pulsar publish from transactionAction.js Change stream already catches the insert event — this was a duplicate source

4.3 Reference Data Cache

Reference collections (platform, bankType, cardSupplier, merchantGateway, merchantGroup, user, playerRank) are small (hundreds to low thousands of records) and change infrequently. Loading them into an in-memory cache eliminates the populate overhead entirely.

CollectionEstimated SizeFields NeededCache TTL
platform ~10-50 records platformId, name 10 min
bankType ~50-200 records bankName 10 min
cardSupplier ~10-50 records name 10 min
merchantGateway ~20-100 records name, code 10 min
merchantGroup ~50-200 records name, accountNumber, gateway 10 min
user ~100-500 records username, name 10 min
playerRank ~10-50 records nameEN 10 min
bankCard Larger, variable maskedCardNumber, accountType, holder, issueBank, merchantPayment 5 min (or on-demand)
Design Note
bankCard is the exception—it may be too large to cache entirely. For this collection, use batch-fetch on demand: collect all unique fromCard IDs from the current batch, fetch them in a single .find({ _id: { $in: ids } }) query, and build a temporary lookup map. This is still far cheaper than per-document populate.

5. Implementation Plan

STEP 1

Create Reference Data Cache

Effort: Medium   Risk: Low

Create a new utility module server/src/helpers/referenceDataCache.js that:

  • Loads all small reference collections into Map<ObjectId, Object> on initialization
  • Auto-refreshes every 10 minutes via setInterval
  • Provides get(collection, id) with fallback to MongoDB on cache miss
  • Provides batchGet(collection, ids[]) for bankCard-style on-demand fetch
  • Exposes warmup() for startup and shutdown() for graceful cleanup
// Pseudocode for referenceDataCache.js

const caches = {
    platform:        new Map(),   // _id → { platformId, name }
    bankType:        new Map(),   // _id → { bankName }
    cardSupplier:    new Map(),   // _id → { name }
    merchantGateway: new Map(),   // _id → { name, code }
    merchantGroup:   new Map(),   // _id → { name, accountNumber, gateway }
    user:            new Map(),   // _id → { username, name }
    playerRank:      new Map(),   // _id → { nameEN }
};

async function refresh() {
    for (const [name, cache] of Object.entries(caches)) {
        const docs = await dbModels[name].find({}).lean();
        cache.clear();
        docs.forEach(doc => cache.set(doc._id.toString(), doc));
    }
}

function get(collection, id) {
    return caches[collection]?.get(id?.toString()) || null;
}

async function batchGet(collection, ids) {
    const missing = ids.filter(id => !caches[collection]?.has(id.toString()));
    if (missing.length > 0) {
        const docs = await dbModels[collection].find({ _id: { $in: missing } }).lean();
        docs.forEach(doc => caches[collection].set(doc._id.toString(), doc));
    }
    return new Map(ids.map(id => [id.toString(), caches[collection].get(id.toString())]));
}

STEP 4

Remove fullDocument: 'updateLookup' from Change Stream

Effort: Low   Risk: Low

In syncClickhousePlugin.js (line 176-178):

Current

const options = {
    fullDocument: 'updateLookup'
};

Forces MongoDB to re-read the full document on every update event.

Proposed

const options = {
    fullDocument: 'whenAvailable'
};

Returns full document only for inserts/replaces (already in oplog). For updates, use updateDescription.updatedFields for filtering.

The change stream filter logic needs adjustment: for update operations, check change.updateDescription.updatedFields.status directly instead of change.fullDocument.status. For insert operations, the full document is already available in the oplog.

STEP 5

Fix Consumer Acknowledgment & Remove Timestamp Hacks

Effort: Low   Risk: Medium

5a. Move acknowledgment to after processing

Current (ack before process)

// Acknowledge ALL immediately
for (const msg of messages) {
    await consumer.acknowledge(msg);
}
// Then process...
await processBatch(batch);

Proposed (ack after process)

try {
    await processBatch(batch);
    for (const msg of batch) {
        await consumer.acknowledge(msg.msg);
    }
} catch (err) {
    for (const msg of batch) {
        await consumer.negativeAcknowledge(msg.msg);
    }
}

5b. Remove generateUniqueTimestamp and fixDuplicateTimestamps

Once ClickHouse tables use stable business keys for dedup (see Section 6), there is no need to manipulate timestamps. Remove:

  • generateUniqueTimestamp() in syncProposals.js
  • fixDuplicateTimestamps() in syncClickhousePluginConsumer.js
  • The while-loop that calls fixDuplicateTimestamps in processBatch()

Replace with plain timestamp formatting:

completeTime: record.completeTime
    ? moment(record.completeTime).format('YYYY-MM-DD HH:mm:ss.SSS')
    : null

6. ClickHouse Schema Changes

Important
This is the most critical change. Without fixing the ORDER BY key, duplicates will persist regardless of other optimizations. This requires table recreation (ClickHouse does not support altering the ORDER BY key of an existing table).

6.1 Migration Strategy

  1. Create new tables with correct schema (e.g., transaction_v2)
  2. Backfill from old tables: INSERT INTO transaction_v2 SELECT ... FROM transaction FINAL
  3. Switch application to write to new tables
  4. Verify data integrity
  5. Rename: drop old table, rename new table

6.2 Dedup Key Design

Transaction: Composite Key Required

A single proposal can generate multiple transaction records with different transactionType values. For example, proposal 10000 may produce:

proposalIdtransactionTypeamount
10000Payment500.00
10000Payment Fee5.00

Using proposalId alone as the ORDER BY key would cause ReplacingMergeTree to merge these into a single row, losing the fee record. The transaction table must use the composite key (proposalId, transactionType).

The other three tables (withdrawal, merchant, deposit) have a 1:1 relationship between proposalId and the record, so proposalId alone is sufficient.

ClickHouse TableORDER BY (Dedup Key)Reason
transaction (proposalId, transactionType) Same proposal has multiple types (Payment, Payment Fee, etc.)
proposalWithdrawal (proposalID) 1:1 — one record per proposal
proposalMerchant (proposalId) 1:1 — one record per proposal
proposalDeposit (proposalId) 1:1 — one record per proposal

6.3 Proposed Table Schemas

transaction

CREATE TABLE transaction_v2 (
    -- ... all existing columns ...
    proposalId     String,
    transactionType String,
    completeTime   Nullable(DateTime64(3)),
    createdAt      DateTime64(3),
    _version       UInt64 DEFAULT toUnixTimestamp64Milli(now64(3))
) ENGINE = ReplacingMergeTree(_version)
PARTITION BY toYYYYMM(createdAt)
ORDER BY (proposalId, transactionType)  -- composite key: same proposal, different types
SETTINGS index_granularity = 8192;

6.4 Query Changes

All report queries that read from these tables should use FINAL to ensure deduplication at query time, or use a subquery pattern: 

-- Option A: FINAL keyword (simpler, slightly slower on large scans)
SELECT * FROM transaction FINAL
WHERE createdAt BETWEEN '2026-01-01' AND '2026-01-31';

-- Option B: argMax subquery (better for large scans)
SELECT argMax(t, _version) AS latest
FROM transaction t
WHERE createdAt BETWEEN '2026-01-01' AND '2026-01-31'
GROUP BY proposalId, transactionType;

8. Risks & Mitigations

RiskSeverityMitigation
Reference cache returns stale data (e.g., platform name changed) Low 10-min TTL limits staleness. Reference data changes are rare and non-critical for reports (name changes are cosmetic). Cache can be manually flushed if needed.
bankCard collection too large to cache Medium Use batch-fetch on demand (per consumer batch), not full cache. Still far cheaper than per-document populate.
ClickHouse table migration causes downtime Medium Use side-by-side migration: create _v2 tables, backfill, switch, verify, then rename. No downtime required.
Removing transaction cron causes gaps if change stream misses events Medium Daily audit job compares counts and re-syncs gaps. Change stream resume tokens in Redis provide recovery on restart.
Ack-after-process causes reprocessing on consumer crash Low ClickHouse ReplacingMergeTree handles idempotent inserts. Reprocessing the same batch produces the same result.
Mapping function changes break existing report queries Medium Output columns and values remain identical. Only the internal resolution method changes (cache vs populate). Verify with diff of sample mapped output.
